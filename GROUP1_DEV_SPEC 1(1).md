## 1. Architecture Overview

Group 1 owns the **auth and customer layer**. Group 2 owns accounts and transactions. The connection between the two teams is the **JWT** — Group 1 issues it, Group 2 validates it on every protected request.

```
Frontend (React/Next.js)
        │
        │  POST /api/auth/register
        │  POST /api/auth/login  ──────► Group 1 issues JWT
        │
        │  Authorization: Bearer <token>
        ▼
  Group 1 (/api/*)          Group 2 (no prefix)
  ─────────────────         ──────────────────────
  /api/auth/register        /customers/{id}         ← uses customerId from G1
  /api/auth/login           /customers/{id}/accounts
  /api/customers            /accounts/{id}
  /api/customers/{id}       /accounts/{id}/deposit
                            /accounts/transfer
```

> **Cross-team dependency:** `customerId` generated by `POST /api/customers` is the
> foreign key for every Group 2 account operation. Both teams must use the same
> `Long` type and auto-increment strategy.

## 2. Domain Model

### 2.1 User entity

```java
@Entity
public class User {
    @Id
    private UUID userId;                  // auto-generated, immutable

    @Column(unique = true, nullable = false)
    private String username;              // email format, lowercase, unique

    @Column(nullable = false)
    private String passwordHash;          // BCrypt — NEVER returned in any response

      private String externalSubjectId; 

      private Long customerId;

    @ElementCollection(fetch = FetchType.EAGER)
    private List<String> roles;           // ["CUSTOMER"] default; ["ADMIN"] elevated

    private boolean isActive;             // true on registration; false = cannot log in

    @Column(updatable = false)
    private Instant createdAt;            // set by @PrePersist

    private Instant updatedAt;            // set by @PreUpdate
}
```

**Field rules:**

| Field | Rule |
|---|---|
| `userId` | UUID, auto-generated, immutable |
| `username` | Valid email format. Unique. Stored as-is (used as login identifier). |
| `passwordHash` | BCrypt. Never stored plain. Never returned in any response, log, or error. |
| `roles` | Defaults to `["CUSTOMER"]`. Only ADMIN may assign `ADMIN` role. |
| `isActive` | Set `true` on registration. If `false`, login is rejected with `403`. |
| `createdAt` | `@PrePersist` — auto-set, immutable. |
| `updatedAt` | `@PreUpdate` — auto-refreshed. |

> **Note for Group 2:** Role vocabulary is `CUSTOMER` and `ADMIN`. The JWT payload
> will carry `roles: ["CUSTOMER"]` or `roles: ["ADMIN"]`. Group 2 must check for
> `CUSTOMER`, not `USER`.

### 2.2 Customer entity

```java
@Entity
public class Customer {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long customerId;              // auto-incremented, cannot be supplied by caller

    @Column(nullable = false)
    private String name;                  // min 2 chars

    @Column(nullable = false)
    private String address;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false)
    private CustomerType type;            // PERSON | COMPANY

    @OneToMany(mappedBy = "customer")
    private List<Account> accounts = new ArrayList<>();  // starts empty; managed by Group 2

    @Column(updatable = false)
    private Instant createdAt;

    private Instant updatedAt;
}
```

**Immutability rule:** `email` and `accountNumber` on a customer are **permanently immutable** after creation. The `PATCH /api/customers/{customerId}` endpoint must inspect the DTO before any DB call and reject the entire request with `400 FIELD_NOT_UPDATABLE` if either field is present — even alongside valid fields.

## 3. JWT Contract

Group 1 signs the JWT. Group 2 validates it. Both services share the same secret.

### Token structure

```json
// Header
{
  "alg": "HS256",
  "typ": "JWT"
}

// Payload (claims)
{
  "sub": "3f2a1b4c-...",      // userId (UUID)
  "roles": ["CUSTOMER"],       // or ["ADMIN"]
  "exp": 1712345678,           // Unix epoch — access token: 3600s; refresh: 604800s
  "iat": 1712342078
}
```

### Implementation

```java
// application.properties (never hardcode the secret here — read from .env)
jwt.secret=${JWT_SECRET}
jwt.access-token-expiry=3600      // seconds
jwt.refresh-token-expiry=604800   // seconds

// Signing
String token = Jwts.builder()
    .setSubject(user.getUserId().toString())
    .claim("roles", user.getRoles())
    .setIssuedAt(new Date())
    .setExpiration(new Date(System.currentTimeMillis() + accessTokenExpiry * 1000L))
    .signWith(Keys.hmacShaKeyFor(secret.getBytes()), SignatureAlgorithm.HS256)
    .compact();
```

### What Group 2 expects

Group 2 extracts claims from the token on every protected request:

- `sub` — maps to local user record to resolve ownership
- `roles` — used for RBAC permission checks (`CUSTOMER` vs `ADMIN`)
- `exp` — validated automatically; expired tokens return `401 SESSION_EXPIRED`

## 4. API Endpoints

All Group 1 endpoints are prefixed with `/api`.

| Method | Endpoint | Auth | Story | Permission |
|---|---|---|---|---|
| `POST` | `/api/auth/register` | None | US-101 | Public |
| `POST` | `/api/auth/login` | None | US-102 | Public |
| `POST` | `/api/auth/password-reset` | None | Future | Public |
| `POST` | `/api/auth/refresh` | Refresh token | Future | — |
| `POST` | `/api/customers` | Bearer JWT | US-103 | `CUSTOMER:CREATE` |
| `PATCH` | `/api/customers/{customerId}` | Bearer JWT | US-104 | `CUSTOMER:UPDATE` |
| `GET` | `/api/customers/{customerId}` | Bearer JWT | US-105 | `CUSTOMER:READ` |

### Authorization filter (applies to all protected endpoints)

```
Incoming request
      │
      ▼
Extract Authorization: Bearer <token>
      │
      ├── Missing / malformed ──► 401 UNAUTHORISED
      │
      ▼
Validate JWT signature + expiry
      │
      ├── Expired ──────────────► 401 SESSION_EXPIRED
      ├── Invalid signature ────► 401 UNAUTHORISED
      │
      ▼
Load local user record by sub claim
      │
      ├── isActive = false ─────► 403 ACCOUNT_INACTIVE
      │
      ▼
Check required permissions
      │
      ├── Missing permission ───► 401 UNAUTHORISED
      │
      ▼
Check ownership (CUSTOMER role only)
      │
      ├── Ownership fails ──────► 401 UNAUTHORISED
      │
      ▼
Process request
```

## 5. US-101 — Register

### Endpoint

```
POST /api/auth/register
Content-Type: application/json
```

### Request body

```json
{
  "username": "lalitha@example.com",
  "password": "Secure@123"
}
```

| Field | Type | Required | Validation |
|---|---|---|---|
| `username` | `String` | Yes | `@Email @NotBlank` — unique in DB |
| `password` | `String` | Yes | `@Pattern` — min 8 chars, 1 uppercase, 1 digit, 1 special char |
| `role` | `String` | No | Optional. Defaults to `CUSTOMER` in service layer. |

### Response — `201 Created`

```json
{
  "userId": "3f2a1b4c-...",
  "username": "lalitha@example.com",
  "roles": ["CUSTOMER"],
  "externalSubjectId": "auth0|abc123",
  "customerId": null,
  "isActive": true,
  "createdAt": "2026-04-07T10:00:00Z"
}
```

> `passwordHash` must **never** appear in this response or any other.

### Service layer logic

```java
// Pseudo-code
public UserResponse register(RegisterRequest req) {
    if (userRepo.existsByUsername(req.getUsername())) {
        throw new ConflictException("USER_ALREADY_EXISTS");
    }
    User user = new User();
    user.setUsername(req.getUsername());
    user.setPasswordHash(passwordEncoder.encode(req.getPassword()));
    user.setRoles(List.of("CUSTOMER"));
    user.setIsActive(true);
    return userRepo.save(user);
}
```

### Error mapping

| HTTP | Error code | When |
|---|---|---|
| `201` | — | Registered successfully |
| `409` | `USER_ALREADY_EXISTS` | Email already in the database |
| `422` | `INVALID_EMAIL_FORMAT` | Not a valid email address |
| `422` | `INVALID_PASSWORD_FORMAT` | Password fails complexity check |
| `422` | `MISSING_REQUIRED_FIELD` | `username` or `password` absent |

### Key test cases

| TC | Scenario | Expected |
|---|---|---|
| TC-101-01 | Valid registration | `201`. `passwordHash` absent from response. |
| TC-101-02 | Duplicate username | `409 USER_ALREADY_EXISTS` |
| TC-101-03 | Weak password | `422 INVALID_PASSWORD_FORMAT` |
| TC-101-04 | Invalid email format | `422 INVALID_EMAIL_FORMAT` |
| TC-101-05 | Missing required field | `422 MISSING_REQUIRED_FIELD` |
| TC-101-06 | `passwordHash` in any response | Must be absent — always |

## 6. US-102 — Login

### Endpoint

```
POST /api/auth/login
Content-Type: application/json
```

### Request body

```json
{
  "username": "lalitha@example.com",
  "password": "Secure@123"
}
```

| Field | Type | Required | Validation |
|---|---|---|---|
| `username` | `String` | Yes | `@Email @NotBlank` |
| `password` | `String` | Yes | `@NotBlank` — compared against BCrypt hash |

### Response — `200 OK`

```json
{
  "accessToken": "eyJ...",
  "refreshToken": "eyJ...",
  "tokenType": "Bearer",
  "expiresIn": 3600
}
```

| Field | Notes |
|---|---|
| `accessToken` | Short-lived (3600s). Include as `Authorization: Bearer <token>` on every protected request. |
| `refreshToken` | Long-lived (604800s). Used at `POST /api/auth/refresh`. |
| `tokenType` | Always `Bearer`. |
| `expiresIn` | Seconds until `accessToken` expires. |

### Service layer logic

```java
public AuthResponse login(LoginRequest req) {
    User user = userRepo.findByUsername(req.getUsername())
        .orElseThrow(() -> new UnauthorisedException("INVALID_CREDENTIALS"));

    if (!passwordEncoder.matches(req.getPassword(), user.getPasswordHash())) {
        throw new UnauthorisedException("INVALID_CREDENTIALS");
        // Do NOT reveal which field was wrong
    }

    if (!user.isActive()) {
        throw new ForbiddenException("ACCOUNT_INACTIVE");
    }

    String accessToken  = jwtService.generateAccessToken(user);
    String refreshToken = jwtService.generateRefreshToken(user);

    return new AuthResponse(accessToken, refreshToken, "Bearer", accessTokenExpiry);
}
```

> **Security rule:** The error for wrong password and for unregistered email must be
> identical (`401 INVALID_CREDENTIALS`). Never disclose which field was the problem.

### Error mapping

| HTTP | Error code | When |
|---|---|---|
| `200` | — | Login successful |
| `401` | `INVALID_CREDENTIALS` | Wrong password or email not found (same response for both) |
| `401` | `SESSION_EXPIRED` | Expired access token used on a protected endpoint |
| `403` | `ACCOUNT_INACTIVE` | Account exists but `isActive = false` |
| `422` | `MISSING_REQUIRED_FIELD` | `username` or `password` absent |

### Key test cases

| TC | Scenario | Expected |
|---|---|---|
| TC-102-01 | Valid credentials, active account | `200`. All 4 token fields present. |
| TC-102-02 | Wrong password | `401 INVALID_CREDENTIALS`. No field hint. |
| TC-102-03 | Unregistered email | `401 INVALID_CREDENTIALS`. Same response as wrong password. |
| TC-102-04 | Correct credentials, inactive account | `403 ACCOUNT_INACTIVE`. |
| TC-102-05 | Expired access token on protected endpoint | `401 SESSION_EXPIRED`. |
| TC-102-06 | No Authorization header | `401 UNAUTHORISED`. |
| TC-102-07 | JWT contains `sub`, `roles`, `exp` | Verified by Group 2 endpoints. |
| TC-102-08 | Missing field | `422 MISSING_REQUIRED_FIELD`. |

## 8. US-104 — Update Customer

### Endpoint

```
PATCH /api/customers/{customerId}
Authorization: Bearer <token>
Content-Type: application/json
```

### Request body

Only fields present in the body are updated. Absent fields stay unchanged.

```json
{
  "name": "Jane Smith",
  "address": "456 New St, Toronto, ON"
}
```

| Field | Type | Required | Behaviour |
|---|---|---|---|
| `name` | `String` | Optional | `@Size(min=2)` if provided |
| `address` | `String` | Optional | No additional constraint |
| `email` | — | **BLOCKED** | Reject entire request → `400 FIELD_NOT_UPDATABLE` |
| `accountNumber` | — | **BLOCKED** | Reject entire request → `400 FIELD_NOT_UPDATABLE` |

> **Immutability rule:** Inspect the DTO **before any DB call**. If `email` or
> `accountNumber` is non-null, reject the entire request immediately — even if
> other fields are valid. No partial updates.

### Response — `200 OK`

```json
{
  "customerId": 42,
  "name": "Jane Smith",
  "address": "456 New St, Toronto, ON",
  "accounts": [],
  "createdAt": "2026-04-07T10:00:00Z",
  "updatedAt": "2026-04-07T11:30:00Z"
}
```

| Field | Notes |
|---|---|
| `customerId` | Unchanged |
| `name` / `address` | Updated value, or original if not in request |
| `accounts` | Unchanged — managed by Group 2 |
| `createdAt` | Unchanged — original creation time |
| `updatedAt` | Refreshed to current UTC by `@PreUpdate` |

### Service layer note

```java
public CustomerResponse update(Long customerId, PatchCustomerRequest req) {
    // 1. Reject immutable fields immediately — before any DB call
    if (req.getEmail() != null || req.getAccountNumber() != null) {
        throw new BadRequestException("FIELD_NOT_UPDATABLE");
    }
    // 2. Load or 404
    Customer customer = customerRepo.findById(customerId)
        .orElseThrow(() -> new NotFoundException("CUSTOMER_NOT_FOUND"));
    // 3. Apply only non-null fields
    if (req.getName() != null)    customer.setName(req.getName());
    if (req.getAddress() != null) customer.setAddress(req.getAddress());
    // updatedAt is handled by @PreUpdate
    return customerRepo.save(customer);
}
```

### Error mapping

| HTTP | Error code | When |
|---|---|---|
| `200` | — | Customer updated |
| `400` | `FIELD_NOT_UPDATABLE` | `email` or `accountNumber` in request body |
| `401` | `UNAUTHORISED` | Missing or invalid token |
| `404` | `CUSTOMER_NOT_FOUND` | No customer with that ID |
| `422` | `INVALID_CUSTOMER_TYPE` | `type` not `PERSON` or `COMPANY` |

### Key test cases

| TC | Scenario | Expected |
|---|---|---|
| TC-104-01 | Update name | `200`. `updatedAt` refreshed. `createdAt` unchanged. |
| TC-104-02 | Update address | `200`. Address updated. |
| TC-104-03 | `email` in body | `400 FIELD_NOT_UPDATABLE`. No changes made. |
| TC-104-04 | `accountNumber` in body | `400 FIELD_NOT_UPDATABLE`. No changes made. |
| TC-104-05 | Valid field + immutable field together | `400`. Entire request rejected. |
| TC-104-06 | Non-existent `customerId` | `404 CUSTOMER_NOT_FOUND`. |
| TC-104-07 | No Bearer token | `401 UNAUTHORISED`. |

## 10. Error Response Schema

All error responses (`400`, `401`, `403`, `404`, `409`, `422`) use this shape:

```json
{
  "code": "USER_ALREADY_EXISTS",
  "message": "A user with this email is already registered.",
  "field": "username"
}
```

| Field | Type | Notes |
|---|---|---|
| `code` | `String` | Machine-readable. Used by frontend and Group 2 for error handling. |
| `message` | `String` | Human-readable. Never expose internal details. |
| `field` | `String` | Optional. Only present when the error maps to a specific request field. |

### Status code semantics

| Code | Use it when |
|---|---|
| `400` | Malformed syntax, type mismatch, invalid path param format, immutable field in body |
| `401` | No valid token, ownership failure, missing permission, CUSTOMER attempts admin-only op |
| `403` | Authenticated but account is inactive |
| `404` | Resource does not exist |
| `409` | Business-state conflict (duplicate email, active accounts blocking delete) |
| `422` | Well-formed but semantically invalid input (bad format, wrong field for type) |

## 11. Security & Guardrails

### Forbidden patterns — never do these

```
✗  Storing passwords in plain text
✗  Returning passwordHash in any response, log, or error message
✗  Hardcoding the JWT secret — use .env / environment variable
✗  Committing .env or any secrets file to version control
✗  Using RestTemplate — use WebClient for any service-to-service calls
✗  Bypassing JPA with raw SQL
✗  Adding fields to a response that are not in the OpenAPI spec
✗  Using AI-suggested dependencies without a security audit
```

### Validation approach

```java
// Controller — let @Valid handle field-level validation
@PostMapping("/register")
public ResponseEntity<UserResponse> register(@RequestBody @Valid RegisterRequest req) { ... }

// Global exception handler
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(MethodArgumentNotValidException ex) {
        String field = ex.getBindingResult().getFieldErrors().get(0).getField();
        String code  = resolveErrorCode(field);
        return ResponseEntity.unprocessableEntity()
            .body(new ErrorResponse(code, "Validation failed", field));
    }

    @ExceptionHandler(ConflictException.class)
    public ResponseEntity<ErrorResponse> handleConflict(ConflictException ex) {
        return ResponseEntity.status(409)
            .body(new ErrorResponse(ex.getCode(), ex.getMessage(), null));
    }
}
```

| Category | Done when |
|---|---|
